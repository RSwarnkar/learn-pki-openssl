# What is a Certificate Signing Request (CSR)?

A **Certificate Signing Request (CSR)** is a file generated by an entity (server, application, user, or Intermediate CA) that requests a trusted Certificate Authority (CA) to issue a digital certificate.

The CSR contains:

* The **Public Key**
* Information about the identity of the requester
* The **digital signature** created using the requester’s **Private Key**

The CSR itself **does not contain the Private Key**.

---

# Why do we need to send a CSR to the Root CA?

In a Public Key Infrastructure (PKI), the **Root CA** is the highest trusted authority.

When an **Intermediate CA** wants a certificate signed by the Root CA, it cannot simply generate the certificate itself and claim trust.
Instead, it must create a CSR and send it to the Root CA for approval and signing.

## Reason

The Root CA owns its own highly protected **Private Key**.

Only the Root CA’s private key can create a valid signature that says:

> “I trust this Intermediate CA.”

This signed certificate establishes a **chain of trust**:

```text
Root CA
   ↓ signs
Intermediate CA
   ↓ signs
Server / User Certificates
```

Because operating systems and browsers already trust the Root CA certificate, they can also trust certificates issued by the Intermediate CA.

---

# What information is sent to the Root CA for certificate signing?

Typically, the following items are sent:

## 1. Certificate Signing Request (CSR)

The CSR contains:

* Public Key
* Common Name (CN)
* Organization Name
* Organizational Unit
* Country
* State
* Locality
* Email Address (optional)
* Requested certificate extensions

Example:

```text
-----BEGIN CERTIFICATE REQUEST-----
MIIC...
-----END CERTIFICATE REQUEST-----
```

---

## 2. Identity Verification Information

Depending on the PKI policy, the requester may also provide:

* Organization verification documents
* Domain ownership proof
* Employee identity proof
* CA authorization documents

For an Intermediate CA, this may include:

* PKI governance approval
* Internal security review
* Key management compliance

---

## 3. Certificate Policy Information (Optional)

The requester may specify:

* Key Usage
* Extended Key Usage
* Certificate validity duration
* Path length constraints
* CA capabilities

Example:

* Can this Intermediate CA issue certificates?
* Can it issue only server certificates?
* Maximum chain depth?

---

# Important Concept

The Root CA does **not** receive the Private Key.

The Private Key must always remain secret and securely stored by the requester.

Only the corresponding **Public Key** inside the CSR is shared.

---

# Basic CSR Flow

```text
1. Generate Private Key
        ↓
2. Generate CSR using the Private Key
        ↓
3. Send CSR to Root CA
        ↓
4. Root CA verifies identity
        ↓
5. Root CA signs certificate using Root Private Key
        ↓
6. Signed certificate is returned
```

---

# OpenSSL Example — Generate CSR

## Generate Private Key

```bash
openssl genrsa -out intermediate-ca.key 4096
```

## Generate CSR

```bash
openssl req -new -key intermediate-ca.key -out intermediate-ca.csr
```

## Contents sent to Root CA

```text
intermediate-ca.csr
```

The private key file:

```text
intermediate-ca.key
```

must NEVER be shared.

---

# Simple Analogy

Think of a CSR like:

> “Here is my identity information and my public key.
> Please verify me and sign my certificate using your trusted authority.”
